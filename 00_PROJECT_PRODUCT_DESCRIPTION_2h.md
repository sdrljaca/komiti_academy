# Product And Project Description: `komiti_academy`

Document Owner: Product Manager
Business Validation Owner: Business Owner
Delivery Coordination: Project Manager

Ово је канонски product/project description документ за пројекат који кандидат прави током onboarding-а.

Сврха овог документа није да објасни како се модул технички имплементира корак по корак, него:
- шта је тачно производ који кандидат гради,
- зашто постоји,
- ко га користи,
- које понашање мора да покрије,
- и како се тај производ постепено materializuje кроз onboarding/curriculum.

`16_CAPSTONE_BUILD_YOUR_OWN_ODOO_MODULE_16h.md` је execution/capstone документ.
Овај документ је product/project description документ.
------------------------------------------------------------------------------------------------------------------

## 1) Шта је `komiti_academy`

`komiti_academy` је Odoo модул за управљање интерним курсевима и њиховим сесијама.

Најлакше је замислити га овако:
- фирма има каталог курсева,
- сваки курс може имати више конкретних извођења,
- свако извођење има инструктора, термин, број мјеста и учеснике,
- систем мора пазити да подаци буду логички исправни.

Примјер:
- курс: `Odoo Basics for New Developers`
- сесија: `Април 2026 / група 1`
- инструктор: један employee
- учесници: више employee-а
- број мјеста: нпр. 12

Кандидат овим модулом не прави само „још два model-а“, него један мали, заокружен business flow.
------------------------------------------------------------------------------------------------------------------

## 2) Пословни problem који модул рјешава

Организација жели да прати:
- који курсеви постоје,
- када се одржавају,
- ко их држи,
- ко похађа коју сесију,
- да ли има слободних мјеста,
- у ком је стању одређена сесија.

Без оваквог модула ти подаци би често били разбацани по:
- spreadsheet-у,
- chat порукама,
- ручним списковима,
- усменом договору.

Циљ `komiti_academy` је да тај flow пресели у уређен Odoo model/view/action/security облик.
------------------------------------------------------------------------------------------------------------------

## 3) Ко су корисници модула

Минимално постоје ова 3 actor-а:

### 3.1) Academy manager

То је особа која:
- креира курсеве,
- креира сесије,
- мијења термине и статусе,
- додаје или уклања учеснике,
- види комплетан преглед.

### 3.2) Instructor

То је особа која:
- држи сесију,
- везана је за конкретне session записе,
- не мора нужно имати пуна администраторска права над цијелим модулом.

### 3.3) Attendee

То је полазник курса.

У првој верзији attendee не мора имати сложен self-service portal.
Довољно је да у data model-у постоји јасна веза да је неки employee/partner пријављен на неку session.
------------------------------------------------------------------------------------------------------------------

## 4) Основни business објекти

Пројекат мора имати бар ова два model-а:

### 4.1) `academy.course`

Ово је „шаблон“ или каталог курса.

Он описује шта је курс као concept, а не једно конкретно извођење.

Примјери:
- `Odoo Basics`
- `Advanced XML View Inheritance`
- `Git and Delivery Discipline`

Минимално треба да носи:
- назив,
- опис,
- ниво тежине,
- active/inactive стање,
- листу својих сесија.

### 4.2) `academy.session`

Ово је конкретно извођење курса.

Примјери:
- `Odoo Basics / 2026-04-10`
- `Git and Delivery Discipline / May group`

Session треба да носи:
- назив,
- везу на course,
- инструктора,
- учеснике,
- почетак и крај,
- број мјеста,
- стање.

Кључна разлика коју кандидат мора разумјети:
- `course` = шта се предаје,
- `session` = када, коме и под којим условима се то конкретно одржава.
------------------------------------------------------------------------------------------------------------------

## 5) Кључни односи

Минимални relation model треба да изгледа овако:
- један `academy.course` има више `academy.session` записа,
- једна `academy.session` припада једном `academy.course`,
- једна `academy.session` има једног инструктора,
- једна `academy.session` има више attendee-а.

Кандидат мора знати да објасни:
- зашто је `course_id` `Many2one`,
- зашто је `session_ids` `One2many`,
- зашто су attendee-и најчешће `Many2many`,
- зашто инструктор није исто што и attendee relation.
------------------------------------------------------------------------------------------------------------------

## 6) Шта корисник мора моћи да уради

Минимални кориснички flow је:

1. Academy manager креира курс.
2. Academy manager отвори курс и креира једну или више сесија.
3. На сесију постави инструктора.
4. На сесију дода attendee-е.
5. Систем израчуна број учесника, слободна мјеста и проценат попуњености.
6. Manager мијења state сесије кроз договорени lifecycle.
7. Корисник може из search/tree/form view-а да пронађе релевантну сесију.

Ако овај flow не ради стабилно, модул није стварно завршен.
------------------------------------------------------------------------------------------------------------------

## 7) Обавезна business правила

Ово није само CRUD пројекат. Модул мора имати бар основна правила која штите data integrity.

Минимално:
- `seats` не смије бити негативан,
- број attendee-а не смије бити већи од броја мјеста,
- инструктор не смије бити уједно attendee на истој сесији,
- `end_date` не смије бити прије `start_date`,
- state мора имати смислен lifecycle.

Препоручени lifecycle у првој верзији:
- `draft`
- `open`
- `done`
- `cancelled`

Кандидат мора моћи да објасни:
- шта је pure data rule,
- шта је UX convenience,
- шта иде у `onchange`,
- шта мора бити server-side гаранција.
------------------------------------------------------------------------------------------------------------------

## 8) Обавезни UI слој

Модул мора бити употребљив у Odoo UI-ју, не само технички постојати у коду.

Минимално треба да постоји:
- course tree view,
- course form view,
- session tree view,
- session form view,
- search view бар за session,
- menu,
- action,
- исправно повезан `view_mode`.

Пожељно, али није обавезно у првој верзији:
- calendar view,
- smart button са курса на сесије,
- notebook organization,
- basic kanban.

Основно правило: ако корисник не може природно да отвори flow кроз menu/action, модул није operationally готов.
------------------------------------------------------------------------------------------------------------------

## 9) Security expectations

У првој верзији није потребан enterprise-level security модел, али није прихватљиво да security уопште не постоји.

Минимум:
- `ir.model.access.csv`,
- бар једна user група,
- бар једна manager група,
- јасно објашњење ко смије create/write/delete.

Прихватљиво је да кандидат каже:
- record rules ће бити phase 2,
- у овој верзији имамо ACL-only приступ,
- разумијем limitation и могу га јасно именовати.

Није прихватљиво да security буде потпуно прећутан.
------------------------------------------------------------------------------------------------------------------

## 10) Шта пројекат учи кандидата

`komiti_academy` је изабран зато што на малом домену тјера кандидата да увеже више Odoo слојева одједном.

Кроз овај пројекат кандидат учи:
- разлику између `_name` и `_inherit`,
- relation modelling,
- computed и related reasoning,
- XML view structure,
- action/menu wiring,
- security basics,
- module upgrade дисциплину,
- runtime verification,
- basic documentation and handoff thinking.

Зато је овај пројекат добар: довољно је мали да не утопи почетника, а довољно богат да открије да ли стварно разумије Odoo.
------------------------------------------------------------------------------------------------------------------

## 11) Шта није циљ прве верзије

Да би scope остао реалан, прва верзија `komiti_academy` не мора да има:
- website portal,
- payment flow,
- invoice integration,
- attendance barcode/check-in,
- certificate generation,
- сложене scheduler/cron automation-е,
- heavy custom JS frontend.

Ово је важно јер кандидат мора научити и шта свјесно оставља out-of-scope.
------------------------------------------------------------------------------------------------------------------

## 12) Acceptance criteria пројекта

Пројекат се сматра успјешно описаним и имплементираним ако кандидат може показати:
- course запис,
- session запис повезан на course,
- instructor-а,
- attendee-е,
- computed поља која дају смислен резултат,
- бар 4 business правила која стварно раде,
- исправан UI flow,
- install и upgrade без traceback-а,
- кратко објашњење architecture-а и ограничења.
------------------------------------------------------------------------------------------------------------------
