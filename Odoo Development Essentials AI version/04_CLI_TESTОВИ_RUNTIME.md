# 04. CLI, тестови и runtime

## Сврха

Ово је дио који раздваја tutorial developer-а од некога ко стварно може испоручити промјену.

Није довољно:
- написати код;
- урадити git commit;
- видјети да server није пао.

Мораш знати како се промјена учитава, тестира и runtime верификује.

Ово је поглавље које student-у често најмање дјелује „интересантно“, а у стварном раду постаје једно од најважнијих. Разлог је једноставан: систем се не квари зато што су људи заборавили синтаксу поља, него зато што нису разумјели шта је стварно учитано, шта је стварно провјерено и шта је само претпоставка.

## CLI као радни алат

Odoo CLI мораш знати бар на основном нивоу:
- install модула;
- upgrade модула;
- stop-after-init режим;
- testing опције;
- shell и дијагностички рад.

Ако избјегаваш CLI, споро напредујеш.

Разлог није у томе што је командна линија „професионалнија“, него у томе што ти она открива шта систем стварно ради. У UI-ју често видиш само крајњи резултат. У CLI-ју видиш install, upgrade, logs, test run и технички контекст. То студенту помаже да престане радити на слијепо.

## Install и upgrade mental model

Почетници често мисле:
- install = довољно заувијек
- restart = довољан за све

То није тачно.

У пракси:
- XML промјене често траже module upgrade;
- Python промјене често траже restart;
- frontend/asset промјене често траже upgrade + restart + hard refresh;
- проблеми након restore-а често траже asset reset дисциплину.

Ово вриједи читати више пута. Велики број Odoo почетничких грешака уопште није у коду, него у погрешној претпоставци шта је потребно да би систем учитао промјену. Student треба да научи да прво класификује врсту промјене, па тек онда одлучује који runtime корак слиједи.

## Тестови

Минимално треба да разумијеш:
- unit test;
- integration test;
- ручни функционални smoke test.

Ово нису конкуренти један другом. Сва три су различита слоја провјере.

То је важна лекција. Почетници често траже један „главни“ доказ да је све добро. У стварности нема једног магичног доказа. Unit test провјерава логички комад понашања. Integration test провјерава шири ток. Ручни smoke test провјерава да систем стварно ради у контексту у ком ће га неко користити. Тек комбинација ових слојева даје реалан ниво повјерења.

## Шта unit tests раде добро

- провјеравају конкретну business логику;
- дају брз feedback;
- штите од regression-а;
- добри су за compute/constraints/method behavior.

## Шта unit tests не рјешавају сами

- да ли је Odoo стварно учитао XML како очекујеш;
- да ли assets раде;
- да ли је action wiring исправан;
- да ли се права корисника у стварном flow-у понашају како мислиш.

## Runtime verification

Ово је пресудно у KomITi начину рада.

Runtime verification значи:
- да је код учитан;
- да је модул upgrade-ован ако треба;
- да је сервер рестартован ако треба;
- да UI flow стварно ради;
- да логови не показују очигледан problem;
- да health/smoke провјера има смисла.

Ово је један од главних зрелосних корака за студента. Кад схватиш runtime verification, престајеш да бркаш „у репоу видим код“ са „систем ради како треба“. Та разлика је пресудна у Odoo-у, јер многе ствари зависе од учитавања модула, asset state-а, базе и циљног окружења.

## Odoo 19 practical note

Што више frontend и web framework тежина расте, то је runtime verification важнији.

У Odoo 19 не можеш више безбрижно мислити да је backend check довољан за коначни доказ исправности.

## Шта провјеравати послије промјене

За backend промјену:
- install/upgrade;
- target flow;
- security case;
- логички исход.

За view/XML промјену:
- upgrade;
- rendered form/list/search;
- action/navigation.

За JS/Owl/asset промјену:
- upgrade;
- restart;
- hard refresh;
- browser/runtime понашање;
- да нема очигледних console/server проблема.

Student може ово посматрати као матрицу. Не постоји један исти post-change поступак за сваку измјену. Добар developer прво зна који је слој промијенио, а онда зна и који доказ мора тражити. То је много бољи приступ него механички радити исту рутину за све.

## Минимална вјежба

1. Направи један Python test за свој модул.
2. Покрени upgrade модула из CLI-а.
3. Отвори циљни flow у UI-ју.
4. Провјери да ли функционалност ради као што test и код обећавају.
5. Објасни разлику између `test је green` и `runtime је verified`.

## Шта мораш знати прије преласка даље

- зашто је CLI важан;
- зашто restart није исто што и upgrade;
- зашто test није исто што и smoke test;
- шта значи runtime verification;
- како различити типови промјене траже различиту провјеру.
