# 03. Видови, action и security

## Сврха

Кад model постоји, још немаш апликацију. Апликацију добијеш тек кад корисник има сигуран и употребљив пут кроз UI.

Ово поглавље је важно јер многи почетници превише дуго гледају само backend. Кад први пут додају model и неколико поља, имају осјећај да су „направили функционалност“. У стварности, корисник ту функционалност још не може ни пронаћи, ни безбједно користити, ни разумно претраживати. Тек овдје систем почиње личити на стварну апликацију.

## UI ланац

У најчешћем случају пут изгледа овако:
- menu
- action
- view modes
- конкретан view
- корисничка интеракција над recordset-ом

Ако ова веза није јасна, почетник обично „крпи“ XML наслијепо.

Добро је да студент овај ланац замисли као пут корисника. Корисник не размишља о model-у, него кликне на menu, уђе у action, отвори одређену листу или форму и онда ради над записима. Ако developer не види систем из корисничке перспективе, правиће технички исправне, али употребно лоше модуле.

## Основни типови view-ова

Мораш знати бар:
- list
- form
- search
- kanban

Пожељно је да разумјеш и:
- calendar
- pivot
- graph
- gantt

Не треба сваки модул све ово, али мораш знати да постоје као дио Odoo UI grammar-а.

Вриједи запамтити да view тип није само питање изгледа. Он носи и начин размишљања. List view служи за рад над више записа, form view за дубљи рад над једним, search view за филтрирање и навигацију, kanban за визуелно праћење стања. Кад student ово разумије, лакше ће одлучивати који UI слој одговара ком business проблему.

## Form view

Form view није само распоред поља. То је мјесто гдје:
- дефинишеш структуру уноса;
- групишеш информације;
- додајеш buttons;
- утичеш на usability;
- откриваш колико добро разумијеш сам business flow.

Ако form view изгледа хаотично, то често значи да и model или business ток нису добро промишљени. Добра форма није само естетика. Она открива шта је важно, шта иде којим редом и које радње корисник стварно има. Зато form view читај као „структуру посла“, не само као распоред widget-а.

## Search view

Почетници је често потцијењују. То је грешка.

Добар search view:
- убрзава рад корисника;
- прави смислене filter-е;
- открива важне business dimensions;
- често је разлика између „модул постоји“ и „модул је употребљив“.

Ово је посебно важно у реалним пословним модулима. Ако корисник не може брзо наћи праве записе, онда функционалност у пракси постаје спора и фрустрирајућа, чак и кад је backend коректан. Зато search view није споредни XML, него дио usability architecture-а.

## Actions

`ir.actions.act_window` је један од најважнијих UI wiring механизама у Odoo-у.

Треба да разумијеш:
- који model action отвара;
- који су `view_mode`-ови;
- какав је context;
- који је domain;
- шта корисник види као default понашање.

Student треба да овдје увиди једну важну ствар: action није само „линк према view-у“. Action одлучује у каквом режиму корисник улази у податке. Да ли одмах види филтриране записе, да ли му је неко поље default-овано, да ли улази у list па онда form, да ли постоји специфичан context. Све то мијења стварно понашање модула.

## Security слој

Security мора доћи рано, не на крају.

Главни елементи су:
- groups;
- access rights;
- record rules;
- explicit checks кад је потребно.

Security је једна од тема гдје student најчешће или претјерано поједностави, или се уплаши. Добар приступ је да га гледа као серију питања: ко уопште смије видјети model, ко смије мијењати записе, који корисник види само своје податке, а који цијели тим. Кад security преведеш на пословна питања, постаје разумљивији.

## ACL

ACL ти одређује ко уопште има create/read/write/unlink приступ model-у.

Ако ACL није постављен како треба, можеш имати два проблема:
- корисник не може ништа да ради;
- још горе, корисник може више него што смије.

Други случај је опаснији, јер систем споља може изгледати „исправно“, а заправо цури приступ или дозвољава погрешне промјене. Зато је боље security провјеравати свјесно, него претпостављати да ће Odoo сам „погодити шта треба“.

## Record rules

Ово је сљедећи ниво security-а. Они не говоре само да ли неко приступа model-у, него којим записима приступа.

Ту domain постаје security механизам.

То је моћно, али опасно ако се ради без јасне business логике.

Ово је мјесто гдје student треба пазити на менталну дисциплину. Record rule није ту да „утјера неки domain који ради“, него да представи пословно правило приступа. Ако то правило не можеш објаснити на јасан, пословни начин, вјероватно га ни не треба још писати без дубљег разумијевања.

## Sudo

`sudo()` је користан, али опасан алат.

Практично правило:
- користи га само ако тачно знаш зашто ти треба;
- не користи га да „брзо поправиш“ security грешку;
- ако си posegao за `sudo()`, мораш знати шта си тиме прескочио.

## View inheritance

Већина озбиљног Odoo рада није greenfield. Чешће ћеш:
- мијењати постојећи form;
- додати поље у туђи list;
- мијењати search filter;
- проширити action/context.

Зато мораш умијети читати и писати стабилан view inheritance.

## Odoo 19 practical note

У Odoo 19 и даље је важно да не мислиш о UI-ју само као о XML markup-у. UI је спој:
- XML architecture;
- action/context/domain wiring-а;
- security правила;
- понекад Owl/frontend extension-а.

## Минимална вјежба

У свом малом модулу:
- додај top menu;
- додај action;
- направи list, form и search view;
- направи бар двије групе корисника;
- дај различит приступ тим групама;
- напиши бар једно record rule правило.

## Шта мораш знати прије преласка даље

- како су menu, action и view повезани;
- зашто је search view важан;
- разлику између ACL и record rules;
- зашто `sudo()` није „стандардно рјешење“;
- основе view inheritance-а.
