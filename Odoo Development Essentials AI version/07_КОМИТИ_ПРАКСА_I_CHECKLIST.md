# 07. KomITi пракса и checklist

## Сврха

Ово поглавље преводи Odoo development из „учио сам framework“ у „могу радити у стварном engineering окружењу“.

Оно је важно јер студент врло лако може стећи лажан осјећај сигурности: прошао је tutorial, направио мали модул и мисли да сада зна development. У стварном тиму то није довољно. Потребно је знати како се ради чиста промјена, како се доказује исправност и како се избјегава operational хаос.

## KomITi mental model

У KomITi-ју није довољно да промјена:
- постоји у git-у;
- прође без syntax грешке;
- изгледа логично у коду.

Мора бити јасно и:
- из ког branch workflow-а долази;
- како се верификује;
- шта је deploy доказ;
- шта је runtime доказ;
- шта је smoke test;
- шта је rollback прича.

Ово је суштина преласка са „учим Odoo“ на „радим у engineering систему“. Student овдје треба да усвоји да квалитет промјене није само у коду, него и у томе да је промјена поновљива, провјерљива и разумљива остатку тима.

## Прије него што почнеш измјену

- знај из ког baseline-а крећеш;
- знај који модул стварно мијењаш;
- знај да ли промјена погађа Python, XML, assets, security или више слојева;
- знај који је минимални доказ исправности.

Ово питање о „минималном доказу“ је пресудно. Ако student у старту не зна шта ће на крају морати доказати, лако потроши вријеме на споредне ствари, а пропусти стварни ризик. На примјер, једна XML измјена не тражи исти доказ као једна security измјена или frontend asset промјена.

## Током измјене

- држи једну логичку цјелину по commit-у;
- не мијешај неповезане измјене;
- читај постојећи код прије override-а;
- пази на security и upgrade посљедице;
- ако је промјена runtime релевантна, планирај verification унапријед.

Ово је добар тренутак да студент развије радну дисциплину. Немој чекати крај да тек тада размишљаш како ћеш тестирати. Добар developer често већ током писања зна шта ће касније морати upgrade-овати, рестартовати, ручно провјерити и шта би могло бити спорно.

## Послије измјене

Постави себи ова питања:
- да ли модул треба upgrade;
- да ли runtime треба restart;
- да ли UI треба hard refresh;
- да ли је smoke test урађен;
- да ли је стварно провјерен захваћени flow;
- да ли документација тражи delta update.

Ово је дио који најлакше испада кад неко учи сам. У solo учењу човјек каже: „ради, завршио сам“. У engineering окружењу то није довољно. Мораш знати и шта је тачно промијењено, како је доказано и да ли неко други може послије наставити од те тачке без нагађања.

## Minimal Definition of Done за Odoo промјену

1. root cause је јасан;
2. код је уредан и у одговарајућем модулу;
3. security импликације су провјерене;
4. upgrade/restart корак је урађен ако треба;
5. runtime flow је тестиран;
6. commit је чист и логички;
7. ако промјена мијења договор тима, документација је ажурирана.

Student треба да чита ову листу као дисциплину затварања задатка. Она служи да не останеш у полузавршеном стању у ком је код ту, али није јасно да ли је безбједан, да ли је учитан, да ли је неко провјерио стварни flow и да ли се тим договорио шта промјена значи.

## Шта junior најчешће пропусти

- заборави upgrade;
- гледа само код, не и runtime;
- не провјери права корисника;
- мисли да је merge исто што и deploy;
- мијеша docs-only и runtime change у исти mental model;
- не разумије шта је минимални operational proof.

Ово није листа за критику, него за самопровјеру. Ако student препозна да редовно пада на некој од ових тачака, то је добар сигнал шта треба систематски поправити у начину рада.

## Завршна checklist

Ако можеш сигурно одговорити на сва ова питања, овај сет ти је дао добру основу:

- Како Odoo модул стварно улази у систем?
- Кад иде install, а кад upgrade?
- Како су menu, action, view и security повезани?
- Кад користиш `compute`, а кад constraint?
- Зашто `sudo()` може бити опасан?
- Шта у Odoo 19 значи Owl/frontend layer?
- Зашто `код постоји` није исто што и `runtime ради`?
- Како изгледа minimum safe delivery discipline у KomITi окружењу?

Ако на нешто од овога одговор није стабилан, ту се треба вратити и поново проћи тему.
