# 01. Подешавање и архитектура

## Сврха

Прије него што пишеш модул, мораш разумјети три ствари:
- гдје Odoo код живи;
- како сервер учитава модуле;
- како backend, база, XML и web client сарађују.

Ако ово није јасно, касније ћеш само механички преписивати pattern-е.

Многи почетници крену обрнутим редом: одмах траже како се дода поље, како се мијења view или како се пише method. То краткорочно даје осјећај напретка, али дугорочно прави проблем, јер developer почне да ради без слике система. Ово поглавље зато служи да изгради основну мапу терена.

## Odoo као систем

Odoo није само Python апликација. То је спој:
- PostgreSQL базе;
- Python backend-а;
- XML data/view слоја;
- web client-а;
- модула који проширују постојећи систем.

Практично, Odoo module development значи да додајеш или мијењаш понашање у једном већ постојећем framework-у.

То је важна разлика у односу на неки „чисти“ Python пројекат. У Odoo-у ријетко градиш систем од празног платна. Чешће улазиш у систем који већ има своја правила, naming, data loading, security и UI grammar. Зато добар Odoo developer прво учи framework, па тек онда пише свој код.

## Основни архитектонски слојеви

### 1. ORM слој

Ту живе:
- model-и;
- fields;
- business logic;
- constraints;
- methods које раде над recordset-овима.

Ово је срце backend-а.

Кад год видиш Odoo model, fields или methods, гледаш у ORM слој. Ту се дефинише шта систем „јесте“: који business entity постоји, која правила важе, које се релације одржавају и како се подаци мијењају. Ако студент не разумије ORM слој, касније ће све остало изгледати као низ насумичних датотека.

### 2. Data слој

Ту живе:
- XML data записи;
- CSV security листе;
- demo data;
- actions, menus, scheduled actions и слично.

Ово није „помоћни“ слој. Велики дио Odoo конфигурације је управо data layer.

Ово је једна од ствари коју почетници тешко прихвате: у Odoo-у XML није само за „изглед“, него често за стварно понашање система. Menu item, action, cron, default records, security записи и велики дио wiring-а живе управо у data слоју. Зато Odoo developer мора читати и Python и XML једнако озбиљно.

### 3. UI слој

Ту живе:
- form, list, search, kanban и друге view архитектуре;
- QWeb templates;
- frontend assets;
- Owl компоненте и web extension тачке.

### 4. Security слој

Ту живе:
- groups;
- access rights;
- record rules;
- понекад field-level ограничења и explicit checks у коду.

Security у Odoo-у није једна датотека. То је систем.

Ово треба читати полако. Почетник често очекује једно „security мјесто“ у којем се све рјешава. У стварности, права зависе од group-а, access rights, record rule логике, понекад explicit checks у коду и од начина како је UI повезан са backend-ом. Зато security у Odoo-у није checkbox, него дио architecture thinking-а.

## Локално окружење

Минимално development окружење за озбиљан рад има:
- Odoo source или јасно дефинисан runtime;
- PostgreSQL;
- custom addons директоријум;
- команду или скрипту за покретање/upgrade;
- developer mode;
- јасан `addons_path`.

У KomITi контексту додатно је важно:
- да знаш из ког Git baseline-а је настао runtime;
- да разликујеш localhost, DEV и PROD;
- да не мијешаш git proof и runtime proof.

Практично, то значи да није довољно да имаш само отворен фолдер у editor-у. Мораш знати и који database користиш, који addons path је активан, да ли је модул стварно учитан, да ли је runtime свјеж и да ли гледаш локално или удаљено окружење. Без тога студент лако дође у ситуацију да „мијења једно, а тестира друго“.

## Шта у Odoo 19 треба посебно пазити

- Odoo 19 docs су сада канонска основа за tutorials и reference.
- Frontend дио се не може свести само на „мало XML-а и мало JS-а“; Owl слој мораш бар разумјети на основном нивоу.
- Asset и web client понашање има већу тежину него што почетници мисле.

## Developer mode

Developer mode је обавезан јер ти открива:
- technical names;
- external ID-еве;
- view inheritance изворе;
- action/menu wiring;
- додатне debug алате.

Без њега си практично слијеп за озбиљнији рад.

Ако студент тек почиње, корисно је да developer mode замисли као режим у ком ти Odoo открива своја техничка имена. Без њега видиш само пословни интерфејс. Са њим почињеш да видиш и framework испод интерфејса. Тек тад можеш повезати оно што кликћеш са model-има, view-овима и action-има који то стоје иза тога.

## Addons path

`addons_path` одређује одакле Odoo учитава модуле.

Практично правило:
- ако модул није на правом `addons_path`-у, Odoo га не види;
- ако је на погрешном `addons_path`-у, можеш имати конфликте или погрешну верзију модула;
- ако не разумијеш `addons_path`, касније ћеш погрешно тумачити зашто модул „не постоји“ или зашто се нека промјена не учитава.

Ово је једна од најчешћих раних замки. Почетник дода код у модул, рестартује систем, а промјене нема. Често проблем није у Python-у, него у томе што Odoo уопште не чита тај модул из очекиваног директоријума. Зато understanding runtime location долази прије debugging-а.

## Како размишљати о модулу

Модул није само фолдер са Python кодом. Модул је пакет који има:
- manifest;
- Python и/или XML/QWeb/JS/SCSS садржај;
- dependency list;
- data loading order;
- security модел;
- upgrade причу.

Добар developer не пита само „како да ово ради“, него и:
- како ће се учитати;
- ко ће смјети користити;
- како ће се надоградити;
- шта ће се десити на постојећим базама.

Ово је први озбиљан mental shift. Почетнички приступ пита: „како да додам функционалност“. Професионални приступ пита: „како да додам функционалност тако да остане стабилна, сигурна и одржива“. Та разлика почиње већ на нивоу architecture thinking-а.

## Најчешће почетничке заблуде

- „Ако сам додао код, Odoo ће га сам видјети.“
- „XML је само UI, није важан као Python.“
- „Security ћу додати касније.“
- „Кад module install прође, све је сигурно исправно.“

Све четири идеје су опасне.

## Минимална вјежба

1. Отвори локални Odoo runtime.
2. Укључи developer mode.
3. Пронађи један постојећи menu item и откриј који `ir.actions.act_window` стоји иза њега.
4. Пронађи један form view и запиши његов external ID.
5. Својим ријечима објасни како су menu, action, model и view повезани.

## Шта мораш знати прије преласка даље

- шта је модул;
- шта је `addons_path`;
- чему служи developer mode;
- који су главни Odoo слојеви;
- зашто runtime и git нису исто.
