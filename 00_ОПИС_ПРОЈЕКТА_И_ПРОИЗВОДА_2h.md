## 1) Опис производа `komiti_academy`

`komiti_academy` је Odoo модул за управљање интерним курсевима и њиховим сесијама.

Најлакше је замислити га овако:
- фирма има каталог курсева,
- сваки курс може имати више конкретних извођења,
- свако извођење има инструктора, термин, број мјеста и учеснике,
- систем мора пазити да подаци буду логички исправни.

Примјер:
- курс: `Odoo Basics for New Developers`
- сесија: `Април 2026 / група 1`
- инструктор: један employee
- учесници: више employee-а
- број мјеста: нпр. 12

Кандидат овим модулом не прави само „још два model-а“, него један мали, заокружен business flow.

### 1.1) Пословни problem који модул рјешава

Организација жели да прати:
- који курсеви постоје,
- када се одржавају,
- ко их држи,
- ко похађа коју сесију,
- да ли има слободних мјеста,
- у ком је стању одређена сесија.

Без оваквог модула ти подаци би често били разбацани по:
- spreadsheet-у,
- chat порукама,
- ручним списковима,
- усменом договору.

Циљ `komiti_academy` је да тај flow пресели у уређен Odoo model/view/action/security облик.

### 1.2) Ко су корисници модула

Минимално постоје ова 3 actor-а:

#### 1.2.1) Academy manager

То је особа која:
- креира курсеве,
- креира сесије,
- мијења термине и статусе,
- додаје или уклања учеснике,
- види комплетан преглед.

#### 1.2.2) Instructor

То је особа која:
- држи сесију,
- везана је за конкретне session записе,
- не мора нужно имати пуна администраторска права над цијелим модулом.

#### 1.2.3) Attendee

То је полазник курса.

У првој верзији полазник не мора имати сложен self-service portal.
Довољно је да у data model-у постоји јасна веза да је неки employee/partner пријављен на неку session.

### 1.3) Основни business објекти

Пројекат мора имати бар ова два model-а:

#### 1.3.1) `academy.course`

Ово је „шаблон“ или каталог курса. Он описује шта је курс као concept, а не једно конкретно извођење.

Примјери:
- `Odoo Basics`
- `Advanced XML View Inheritance`
- `Git and Delivery Discipline`

Минимално треба да носи:
- назив,
- опис,
- ниво тежине,
- active/inactive стање,
- листу својих сесија.

#### 1.3.2) `academy.session`

Ово је конкретно извођење курса.

Примјери:
- `Odoo Basics / 2026-04-10`
- `Git and Delivery Discipline / May group`

Session треба да носи:
- назив,
- везу на course,
- инструктора,
- учеснике,
- почетак и крај,
- број мјеста,
- стање.

Кључна разлика коју кандидат мора разумјети:
- `course` = шта се предаје,
- `session` = када, коме и под којим условима се то конкретно одржава.

### 1.4) Кључни односи

Минимални relation model треба да изгледа овако:
- један `academy.course` има више `academy.session` записа,
- једна `academy.session` припада једном `academy.course`,
- једна `academy.session` има једног инструктора,
- једна `academy.session` има више attendee-а.

Кандидат мора знати да објасни:
- зашто је `course_id` `Many2one`,
- зашто је `session_ids` `One2many`,
- зашто су attendee-и најчешће `Many2many`,
- зашто инструктор није исто што и attendee relation.

### 1.5) Шта корисник мора моћи да уради

Минимални кориснички flow је:
1. Academy manager креира курс.
2. Academy manager отвори курс и креира једну или више сесија.
3. На сесију постави инструктора.
4. На сесију дода attendee-е.
5. Систем израчуна број учесника, слободна мјеста и проценат попуњености.
6. Manager мијења state сесије кроз договорени lifecycle.
7. Корисник може из search/tree/form view-а да пронађе релевантну сесију.

Ако овај flow не ради стабилно, модул није стварно завршен.

### 1.6) Обавезна business правила

Ово није само CRUD пројекат. Модул мора имати бар основна правила која штите data integrity.

Минимално:
- `seats` не смије бити негативан,
- број attendee-а не смије бити већи од броја мјеста,
- инструктор не смије бити уједно attendee на истој сесији,
- `end_date` не смије бити прије `start_date`,
- state мора имати смислен lifecycle.

Препоручени lifecycle у првој верзији:
- `draft`
- `open`
- `done`
- `cancelled`

Кандидат мора моћи да објасни:
- шта је pure data rule,
- шта је UX convenience,
- шта иде у `onchange`,
- шта мора бити server-side гаранција.

### 1.7) Обавезни UI слој

Модул мора бити употребљив у Odoo UI-ју, не само технички постојати у коду.

Минимално треба да постоји:
- course tree view,
- course form view,
- session tree view,
- session form view,
- search view бар за session,
- menu,
- action,
- исправно повезан `view_mode`.

Пожељно, али није обавезно у првој верзији:
- calendar view,
- smart button са курса на сесије,
- notebook organization,
- basic kanban.

Основно правило: ако корисник не може природно да отвори flow кроз menu/action, модул није operationally готов.

### 1.8) Security expectations

У првој верзији није потребан enterprise-level security модел, али није прихватљиво да security уопште не постоји.

Минимум:
- `ir.model.access.csv`,
- бар једна user група,
- бар једна manager група,
- јасно објашњење ко смије create/write/delete.

Прихватљиво је да кандидат каже:
- record rules ће бити phase 2,
- у овој верзији имамо ACL-only приступ,
- разумијем limitation и могу га јасно именовати.

Није прихватљиво да security буде потпуно прећутан.

### 1.9) Шта пројекат учи кандидата

`komiti_academy` је изабран зато што на малом домену тјера кандидата да увеже више Odoo слојева одједном.

Кроз овај пројекат кандидат учи:
- разлику између `_name` и `_inherit`,
- relation modelling,
- computed и related reasoning,
- XML view structure,
- action/menu wiring,
- security basics,
- module upgrade дисциплину,
- runtime verification,
- basic documentation and handoff thinking.

Зато је овај пројекат добар: довољно је мали да не утопи почетника, а довољно богат да открије да ли стварно разумије Odoo.

### 1.10) Шта није циљ прве верзије

Да би scope остао реалан, прва верзија `komiti_academy` не мора да има:
- website portal,
- payment flow,
- invoice integration,
- attendance barcode/check-in,
- certificate generation,
- сложене scheduler/cron automation-е,
- heavy custom JS frontend.

Ово је важно јер кандидат мора научити и шта свјесно оставља out-of-scope.

### 1.11) Acceptance criteria пројекта

Пројекат се сматра успјешно описаним и имплементираним ако кандидат може показати:
- course запис,
- session запис повезан на course,
- instructor-а,
- attendee-е,
- computed поља која дају смислен резултат,
- бар 4 business правила која стварно раде,
- исправан UI flow,
- install и upgrade без traceback-а,
- кратко објашњење architecture-а и ограничења.

## 2) Опис пројекта `komiti_academy`

Пројекат `komiti_academy` у овом curriculum-у није само крајњи модул као производ, него и јасан редослијед кандидатских задатака, референци и контролних тачака кроз које тај модул постепено настаје.

### 2.1) Пројектни план
Као што је у `00_END2END_ONBOARDING_ПУТ_УЧЕЊА_39-52w.md` речено, за кандидата без претходног програмерског искуства реалан распон је; овдје је тај исти временски оквир приказан кроз scope који треба да буде испоручен у свакој фази:
- **Junior: Основна оперативна самосталност:** 12 седмица у стандардном ритму; 8–10 седмица само у убрзаном режиму
	1.1) Фаза A — Foundations (седмице 1–4): испорука основног разумијевања рада, локалног окружења, Odoo основа и првог `komiti_academy` skeleton-а.
	1.2) Фаза B — Guided Odoo Development (седмице 5–8): испорука installable MVP-а са model-има, relation-има, view-овима, action/menu слојем и basic security.
	1.3) Фаза C — Independent Delivery (седмице 9–12): испорука business rules, verification note-а, docs delta и јасног Definition of Done status-а за `komiti_academy`.
- **Mid-level рад на пројекту:** 4–8 мјесеци
	1.4) Рани и средњи дио Фазе D — Professional Growth (мјесеци 4–8): испорука стабилнијег, чистијег и review-safe модула који кандидат може самостално одржавати, проширивати и бранити кроз review.
- **Senior: Сам води сложеније измјене end-to-end:** 9–12 мјесеци
	1.5) Каснији дио Фазе D — Professional Growth (мјесеци 9–12): испорука зреле верзије `komiti_academy` као личног доказа да кандидат умије водити сложенији scope, rollout и handoff готово без надзора.

### 2.2) Задаци у пројекту
Док читаш learning library, различити дијелови `komiti_academy` постају конкретни кроз различите документе, а конкретни пројектни задаци за кандидата се налазе на крају сваког релевантног документа у поглављима `99) Задатак на komiti_academy пројекту за кандидата` и `99) Рјешења`.

Ти задаци су сада постављени у јединственој форми: нумерисани су, испод сваког task-а стоји `Референца` на поглавља у којима је тема објашњена, а у `99) Рјешења` исти број task-а прати јасно разложен корак-по-корак поступак.

Кад желиш једним читањем да видиш све task-ове који воде ка готовом `komiti_academy`, користи ове референце:

- `02_БРЗИ_ПОЧЕТАК_РАДА_1h.md`:  project-specific почетни задатак и како да га припремиш кроз runtime, health и search секције.
- `03_GIT_VSCODE_ОСНОВЕ_2h.md`: Git task-ови за feature branch, clean commit и push discipline, са упутом из workflow поглавља.
- `04_ИНФРА_TERRAFORM_DOCKER_AWS_ОСНОВЕ_4h.md`: runtime/infra reasoning task-ови, локални Terraform + Docker Desktop lab за `komiti_academy` dev runtime и упута како да разликујеш infra enablement од application проблема и core product scope-а.
- `05_ODOO_ОСНОВЕ_4h.md`: први стварни project task-ови за почетну структуру модула, model-е, relation-е, поља и UI, са упутом на foundations поглавља.
- `06_АНАТОМИЈА_ДОБРОГ_ODOO_МОДУЛА_2h.md`: task-ови за module structure и упута која поглавља воде manifest, XML и security провјеру.
- `07_КАКО_СЕ_ЧИТА_ODOO_МОДУЛ_3h.md`: task-ови за читање властитог модула и упута којим редослиједом то радиш.
- `10_ODOO_ПРИРУЧНИК_ЗА_ТЕСТИРАЊЕ_И_ПРОВЈЕРУ_3h.md`: verification task-ови и јасна референца на testing, role-based testing и поглавља о провјери завршног пројекта.
- `12_ODOO_ЛИСТА_ПРОВЈЕРЕ_ЗА_ПРЕГЛЕД_КОДА_1h.md`: сопствена review провјера и упута како да прођеш review checklist корак по корак.
- `13_РАЗЛАГАЊЕ_ЗАДАТКА_И_РАЗМИШЉАЊЕ_О_ИСПОРУЦИ_2h.md`: planning/risk task-ови и директна упута на planning template и risk questions.
- `16_ЗАВРШНИ_ПРОЈЕКАТ_НАПРАВИ_СВОЈ_ODOO_МОДУЛ_16h.md`: главни implementation task-ови и најдиректнији step-by-step пут кроз scope, rules, UI, security, checkpoints и DoD.

`komiti_academy` је мали, али заокружен Odoo модул за управљање курсевима и њиховим сесијама, изабран тако да кандидат кроз један реалан domain flow научи model, relation, view, security, upgrade и verification дисциплину end-to-end.
